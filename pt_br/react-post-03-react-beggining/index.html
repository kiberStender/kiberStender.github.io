Se você não está muito bem familiarizado com Javascript irá dizer que o primeiro pedaço é mais rápido, porque tem apenas 21 itens. A questão aqui é: innerHTML não é uma simples variável. Qualquer alteração à ela dispara operações de I/O(Input/Output)(Entrada/Saída) em seu navegador. I/O é uma operação cara e não somente isso mas diferente de quando se altera o valor de uma variável, fazer I/O pode falhar devido a muitos possiveis problemas(como ter um atraso no proceso devido a falta de recursos como cpu e memória no momento e o navegador cancelar sua operação de I/O) e dada a decisão de fazer como uma variável ao invés de uma função, não é visível ao desenvolvedor que isso poderá falhar e erros podem acontecer por desatenção.

Ao executar o primeiro trecho de código nós estamos disparando 21 operações de I/O enquanto que no segundo trecho, apesar de termos um loop maior (101 itens) nós somente fazer operação de I/O.

E o que o React tem haver com isso?

React trabalha com a idéia de controlar o número de operações IO que você faz ao não te permitir fazer isso. Mas como o React sabe quando fazer I/O e quando não fazer? React faz comparações de variáveis, logo a qualquer momento que uma variável muda algo React dispara uma re-renderização em somente as partes de seu código que são afetadas por aquele valor que mudou. Para controlar isso, React quer que você não mude variáveis diretamente:

let a = 10
a = 5 // Não faça isso

React tem uma 'interface' onde tem uma função/método de nome setState e ao usar esta função, você controla a renderização or deixa o React saber quando a variável mudou e disparar a operação re-renderização(mais desse assunto depois). Ao fazer isso, React sempre sabe quando e quantas operações de I/O a serem executadas, porque ele controla o que ele chama de Componentes e quaisquer Componentes dentro do Componente que disparou o setState serão atualizados também.

Ao invés de fazer um I/O por Componente ele faz o que a função reduce faz em um array, processando tudo em uma única leva de processamento retornando ao fim um único object para o React fazer o I/O.

JSX

Espere um minuto... O que é essa síntaxe estranha? Isto provavelmente não é o que você viu até então quando as pessoas falavam de React. O que você provavelmente viu é isso:

import React from "react";

export default class Greetings extends React.Component {
  render() {
    return <div id="greetings-component" className="a-class-that-changes-font-to-red">Hello Mr/Ms {this.props.name}?</div>
  }
}

Como obter isso? Esta síntaxe HTML no meio do seu Javascript é chamada JSX. É uma linguagem trsnspilada no meio do seu Javascript transpilado que é transformada nesta 'estranha' síntaxe que você viu acima. Para ter essa síntaxe em seu código tornando mais fácil escrever e ler, você precisa somente adicionar outro 'preset' na sua configuração do Babel (no arquivo webpack.config.js). Primeiro instale o preset do react do Babel:

npm install --save-dev @babel/preset-react@7.9.4

Agora vá até webpack.config.js e adicione o recém instalado preset à nossa configuração:

...
options: {
  presets: ['@babel/preset-env', '@babel/preset-react'] // Apenas adicione '@babel/preset-react'
}
...

PS: Toda vez que você ver três pontos ..., isso significa: "mantenha o que vier antes disso". Ou se vier depois do que estamos a adicionar/modificar, significa: "mantenha o que estiver após os três pontos".

Agora você pode começar a escrever JSX HTML dentro de seus arquivos Javascript e o Babel irá transpilar corretamente.

  1. app.js

  <Greetings name="John Titor" />,

React.createElement e JSX

Entender como o JSX é transformado em React.createElement pode nos ajudar a entender algumas informações que veremos com mais detalhes nos próximos tópicos. Primeiro veremos o protótipo de createElement:

React.createElement(
  tipo,
  [props],
  [...filhos]
)
// Using it
React.createElement(
  'div',
  {prop1:"string", prop2:uma_variavel},
  "filho1", "filho2", React.createElement('span', {id:"abc"}, 'texto')
)

React.createElement(
  Greetings,
  {nome:"Dev"},
  "filho1", "filho2", React.createElement(UmComponente, {id:"abc"})
)
  1. tipo: Uma String ou objeto que contém o component que estamos a instanciar. Para qualquer Tag nativa de HTML, você o passa como uma String assim como 'div', 'span', 'article', etc. Para seus próprios Componentes como nosso Greetings, você o passar não como uma String mas como uma 'definição de classe anônima' ou um tipo se preferir
  2. props: É um objeto literal {} contendo os valores que você quer passar para seu Component instanciado. No protótipo está envolto de [] não porque é um array, mas para dizer que é opcional
  3. filhos: É um array vargas que você passa o que você quiser, contanto que seja serializável para String. Geralmente Componentes customizados não têm filhos. Não é uma regra, é apenas que normalmente não se aninha coisas dentro de seu Componente de fora do arquivo que o define. No protótipo está envolto de [] não porque é um array, mas para dizer que é opcional

Agora a representação JSX:

<div prop1="string" prop2={uma_variavel}>
  filho1
  filho2
  <span id="abc">texto</span>
</div>

<Greetings name="Dev">
  filho1
  filho2
  <UmComponent id="abc"/>
</Greetings>

Componentes do React

Eu disse anteriormente que o React tem algo chamado Componente. Mas o que é um Componente: Um Componente pode ser duas coiasas:

  1. Uma função que retorna JSX/React.createElement:
    //1
function Greeter(props){
  return <div id="greeter-component">{props.greet} {props.treatment} {props.name}</div>

//ou
//2
function Greeter(props){
  return React.createElement('div', {id:"greeter-component"}, `${props.greet} ${props.treatment} ${props.name}`);
}
    ou

const Greeter = (props) => <div id="greeter-component">{props.greet} {props.treatment} {props.name}</div>
//ou
//4
const Greeter = (props) => React.createElement('div', {id:"greeter-component"}, `${props.greet} ${props.treatment} ${props.name}`);
  2. Uma classe que herda de React.Component ou React.PureComponent e tem um método render que retorna JSX/React.createElement
    class Greeter extends React.Component {
  constructor(props){
    super(props)
  }
  render() {
    return <div>{this.props.greet} {this.props.treatment} {this.props.name}</div>

//or
class Greeter extends React.Component {

  }
}
    PS: Se você não está a adicionar nada além do chamado de super em seu construtor, é opcional escrevê- lo:
    class Greeter extends React.Component {
  render() {
    return <div>{this.props.greet} {this.props.treatment} {this.props.name}</div>
  }

Componente feito em Function vs feito Classe

A vantagem em ter um Componente feito em uma função é que, torna- se menos verboso e rápido de ser escrito, veja os exemplos 1, 2, 3 e 4(especialmente 3 e 4). Geralmente você irá usar desta forma quando tiver um simples Componente, como nosso Greet, que somente mostra alguma coisa e não possui interações com o usuário como quando se cria uma Cabeçalho, Rodapé ou qualquer parte estática de sua aplicação que não irá mudar por ações internas do próprio Componente.

Por outro lado ter um Componente como uma classe, irá lhe garantir mais controle sobre seu Componente, pois você terá suporte dos ciclos de vida do React e mudança de estados usando this.setState. Você perde na quantidade de code escrito, mas ganha no nível de melhorias e otimização de seu Componente.

De qualquer forma, para mudar de classe para função ou vice-versa não é tão difícil, então se você não está seguro qual a melhor abordagem a se usar, vá com a abordagem de classe, se você ver que seu Componente é apenas mudar por fora(via props), ou você não precisar melhorar/usar ciclos de vida, você pode facilmente mudar para a abordagem de função depois.

O que são esses "props"?

Props é a abreviação de "Propriedades". Pense em seu Componente como se fosse uma função, os props são seus parâmetros, mas ao invés de passà los por posição como você normalmente faz em uma função, aqui você usa os nomeia para passá- los em qualquer posição.

Uma vez que você passa um props em seu JSX ou React.createElement você poderá acessá- lo dentro de sua classe/função que declarou seu Componente usando props.[o exato mesmo nome] que você declarou fora da classe/função quando instanciou- o. Você pode passar literalmente qualquer coisa via props: funções, strings, int, double, objetos complexos, qualquer coisa que o javascript permite vocẽ por em um objeto, porque se você se lembrar como passamos props na função createElement, é justamente um objeto literal. Instanciando

Observe que props.greet na primeira iamgem está a ser populaddo na segunda imagem via greet="Hello". Se você esquecer de popular desta forma, seu props.greet terá como valor undefined em sua classe/função. Logo, lembre-se sempre de ou proteger seus props com algum if else ou sempre popule seus props corretamente quando instanciar seu Componente. Deste jeito, quem quer que esteja a usar seu Componente Greeter e esquecer de popular greet, o Componente irá assumir que o valor é "Hello".

So the state is ... ?

Enquanto os props são valores extermnos passados a seu Component via um Componente pai, eles são complementante imutáveis e mudá- los dentro de seu Componente não irá disparar nenhuma re-renderização. Logo como podemos mutar valores para interagir como nossos usuários? Como mencionado no ínicio, React tem uma função de nome setState para lidar com mutações que somente estão disponíveis se seu Componente é uma classe que herda de React.Component ou React.PureComponent. Você precisa criar uma variável de nome state(e tenha atenção para não dar outro nome, tem que ser state) e esta variável deverá ser um objeto contento, se possívelm apenas valores (como String, Int, Double, classes complexas represetando um valor como Date) ou qualquer coisa que pode ser comparável:

...
constructor(props){
  super(props)
  this.state = {
    greet: "",
    treatment: "",
    title: "",
    aNumber: 0
  }
}
...

E para mudar um dos valores que você tem você usa setState deste forma:

...
minhaFuncaoParaMudarGreet(newGreet){
  this.setState({greet: newGreet})
}
...

Ou:

...
minhaFuncaoQueIncrementa(value){
  this.setState(state => ({aNumber: state.aNumber + value}))
}
...

Ou:

...
minhaFuncaoQueMudaGreetEIncrementaANumber(newGreet, aNumberToIncrement){
  this.setState((state, props) => ({aNumber: state.aNumber + aNumberToIncrement, greet: newGreet}))
}
...

Independente do que você escolher(passar objeto simples ou passar função anônima) este comando irá disparar o React para comparar o antigo valor de greet com newGreet e aNumber com o novo valor para checar se há diferenças. Se houverem diferenças, React re-renderizará seu Componente e todos os filhos dentro dele. Atente-se a esta parte React re-renderizará seu Componente e todos os filhos dentro dele. Então se você não quebrar sua aplicação em tantos pequenos Componentes quanto puder, você poderá ter problemas de performance por ter muitas coisas para re-renderizar toda vez que mudar uma única variável.

A mudar treatment

Todos esses exemplos de como mudar valores dentro de this.state são maravilhosos mas, como realmente se faz o mundo real? Vamos adicionar um input para permitir que o usuário digite o que quiser que seja mostrado no "campo treatment" ao invés de "Mr/Ms". Inputs em React são Componentes simples que exigem que você faça o bind/attachment/implementação do listener onChange e opcionalmente faça o bind/attachment de uma variável no props value. Bind/Attach são os termos usados para descrever a ação de passar uma variável para algum prop em um dado Componente, neste caso um input. Este listener onChange é apenas uma variável com a assinatura:

onChange(event: Event): void

Vamos modificar nosso src/javascript/greetings/Greetings.js um poucopara que você tenha um exemplo:

...
  constructor(props){
    super(props);
    this.state = {
      treatment: ""
    }
  }
  
  whenTreatmentChanges(newValue){
    this.setState({treatment: newValue})
  }
  
  render(){
    return (
      <div id="greetings-component" className="a-class-that-changes-font-to-red">
        <div className="row">
          <div className="col-1">Treatment</div>
          <div className="col-2">
            <input value={this.state.treatment} onChange={event => this.whenTreatmentChanges(event.target.value)} />
          </div>
        </div>
        <Greeter greet="Hello" treatment={this.state.treatment} name={this.props.name} />
      </div>
    )
  }
...

setState callback

setState é uma função assíncrona, então se você quiser mudar algo uma linha abaixo da chamada da mesma esperando que já terá os novos valores isso não funcionará. Logo se você precisa disparar alguma coisa exatamente quando setState terminar de atualizar sua variável state, você poderá oassar uma função anônimo como segundo parâmettro de setState:

...
whenTreatmentChanges(newValue){
  this.setState({treatment: newValue})
  console.log(this.state.treatment) // printa o valor antigo de treatment
}
...

Isto irá mostrar o valor antigo porque como explicado, setState é assíncrona, então para corretamente fazer algo após setState terminar de atualizar você precisa fazer da seguinte forma:

...
whenTreatmentChanges(newValue){
  this.setState({treatment: newValue}, () => console.log(this.state.treatment)) // printa o valor atualizado de treatment
}
...
